# 贪吃蛇 OJ 版自动策略程序：算法方案文档

## 1. 任务目标

需要编写一个 **C 语言交互式程序**，控制一条贪吃蛇在 20×20 地图中自动移动，尽可能多吃食物并避免死亡。

地图元素含义如下：

```text
#  墙，不可穿越
.  空地
H  蛇头
B  蛇身
F  食物
O  障碍物
```

地图大小固定为 `20 × 20`，边缘是墙，障碍物固定不动。蛇初始长度为 3，包含蛇头。食物初始有 1 个，每吃掉一个食物后，交互程序会立即给出新的食物坐标。蛇每吃一个食物加 10 分。蛇还会每走 `N` 步自然增长一节；如果第 `N` 步正好吃到食物，也只增长一节，不增长两节。增长方式是在尾部增长，即尾巴原位置不清空。

程序每一步输出两行：

```text
方向字符：W/A/S/D
当前移动前的分数
```

每次输出后必须：

```c
fflush(stdout);
```

然后读取交互程序返回的两个整数：

```text
r c
```

含义：

```text
1~18 范围内：新食物坐标
20 20：交互正常继续，没有新食物
100 100：游戏结束，需要输出当前地图和当前得分，然后结束程序
```

注意：如果收到 `100 100`，要求输出的是 **碰撞前一刻的地图和得分**。

---

## 2. 总体策略

采用一个中等复杂度、可落地、比纯 BFS 更稳的策略：

```text
候选路径搜索 + 蛇移动模拟 + 局面评分函数
```

核心思想：

1. 当前食物位置已知；
2. 先用 BFS 求蛇头到食物的最短距离 `d`；
3. 再用 DFS / 回溯枚举若干条长度不超过 `d + DELTA` 的安全候选路径；
4. 对每条路径完整模拟蛇的移动，包括：
   - 蛇头移动
   - 蛇身跟随
   - 吃食物增长
   - 每 N 步自然增长
   - 同一步吃食物和自然增长只增长一节
5. 得到吃完当前食物之后的新状态；
6. 用评分函数评价这个状态的未来安全性；
7. 选择评分最高的候选路径，执行它的第一步；
8. 如果没有可吃路径，则执行保命策略：追尾巴 / 走向最大可达空间。

这个方案不是精确求全局最优，因为未来食物随机刷新，完整搜索会爆炸。它采用有限候选路径和启发式评分近似未来期望。

## 3. 状态表示

### 3.1 坐标结构体

```c
typedef struct {
    int r;
    int c;
} Point;
```

### 3.2 蛇结构

建议用数组维护蛇，而不是只依赖地图。

```c
Point snake[400];
int len;
```

约定：

```text
snake[0]       是蛇头
snake[len - 1] 是蛇尾
```

### 3.3 方向

```c
// W A S D
int dr[4] = {-1, 0, 1, 0};
int dc[4] = {0, -1, 0, 1};
char dirChar[4] = {'W', 'A', 'S', 'D'};
```

方向编号：

```text
0 = W
1 = A
2 = S
3 = D
```

判断反向移动：

```c
int isOpposite(int d1, int d2) {
    return (d1 == 0 && d2 == 2) ||
           (d1 == 2 && d2 == 0) ||
           (d1 == 1 && d2 == 3) ||
           (d1 == 3 && d2 == 1);
}
```

需要记录当前方向：

```c
int curDir;
```

初始方向可以根据蛇头和相邻蛇身推断。

例如：

```text
如果 snake[1] 在蛇头左边，说明蛇当前向右；
如果 snake[1] 在蛇头右边，说明蛇当前向左；
如果 snake[1] 在蛇头上方，说明蛇当前向下；
如果 snake[1] 在蛇头下方，说明蛇当前向上。
```

## 4. 地图初始化

读入前 20 行地图和第 21 行 `N`。

需要从地图中找到：

```text
H：蛇头
B：蛇身
F：食物
O：障碍物
#：墙
```

难点是：地图上只标了 `B`，没有直接给出蛇身顺序。

初始蛇长度固定为 3，所以可以这样恢复蛇数组：

1. 找到 `H`；
2. 在 `H` 四周找相邻的 `B`，它一定是 `snake[1]`；
3. 再在 `snake[1]` 四周找另一个 `B`，它是 `snake[2]`；
4. 不要回到 `snake[0]`。

伪代码：

```text
snake[0] = H

for 四个方向:
    如果 H 邻居是 B:
        snake[1] = 这个位置

for 四个方向:
    如果 snake[1] 邻居是 B 且不是 snake[0]:
        snake[2] = 这个位置

len = 3
```

然后根据 `snake[0]` 和 `snake[1]` 推断当前方向。

## 5. 蛇移动模拟

这是整个程序最核心的函数。

### 5.1 输入输出

设计函数：

```c
int simulateMove(GameState *s, int dir);
```

含义：

```text
在状态 s 上执行 dir 方向移动一格。
如果移动合法，更新状态并返回 1。
如果死亡，状态不应破坏，返回 0。
```

为了方便候选路径模拟，建议定义完整状态：

```c
typedef struct {
    char map[20][21];
    Point snake[400];
    int len;
    int score;
    int step;
    int curDir;
    int foodR;
    int foodC;
} GameState;
```

注意 `map[20][21]` 是为了每行末尾放 `\0`。

### 5.2 移动规则

一次移动过程：

```text
1. 检查是否反向移动；
2. 计算新头位置；
3. 判断新头是否撞墙、障碍、身体；
4. 判断是否吃到食物；
5. 判断是否自然增长；
6. grow = eatFood || naturalGrow；
7. 更新蛇数组；
8. 更新地图；
9. 更新分数、步数、方向。
```

增长判断：

```c
int nextStep = s->step + 1;
int eatFood = (newR == s->foodR && newC == s->foodC);
int naturalGrow = (nextStep % N == 0);
int grow = eatFood || naturalGrow;
```

题目规定：吃食物和自然增长同一步只增长一节，所以用 `||`。

### 5.3 尾巴特殊判断

普通移动时，尾巴会离开原位置，因此蛇头可以走到“当前尾巴的位置”。

但增长时，尾巴不会离开原位置，因此蛇头不能走到当前尾巴。

所以判断撞身体时要区分：

```text
如果 grow == 0：
    当前尾巴格子可以视为即将空出来
如果 grow == 1：
    当前尾巴格子仍然被占用
```

合法性判断：

```text
新头不能是 # 或 O
新头不能撞 B
但如果新头等于当前尾巴位置，并且本步不增长，则允许
```

## 6. BFS 最短距离

### 6.1 用途

BFS 用来计算：

```text
蛇头到当前食物的最短距离 d
蛇头到蛇尾的距离
某个状态下可达空间大小
```

### 6.2 BFS 规则

在某个静态状态下，把不可走格子视为：

```text
# 墙
O 障碍物
B 蛇身
```

可走格子：

```text
.
F
尾巴位置可根据场景选择是否允许
```

对于估算距离，通常可以允许走到蛇尾，因为尾巴未来会移动。

但对于“立即下一步是否合法”，必须严格用 `simulateMove` 判断。

## 7. 候选路径生成

### 7.1 为什么不能只用标准 BFS？

标准 BFS 只能返回一条最短路或最短距离，不能自然返回第 2、第 3、第 4 短路径。

而本策略需要比较多条候选路径，因为：

```text
最短路径不一定最安全。
绕远几步可能保留更多空间。
```

所以采用：

```text
BFS 求最短距离 d
DFS 枚举长度 <= d + DELTA 的若干条安全路径
```

### 7.2 参数建议

```c
#define DELTA 4
#define MAX_CANDIDATES 30
#define MAX_PATH_LEN 80
```

含义：

```text
只枚举比最短路多 DELTA 步以内的路径
最多保留 MAX_CANDIDATES 条候选路径
```

初期可以设：

```text
DELTA = 3 或 4
MAX_CANDIDATES = 20 或 30
```

不要过大，否则路径数量会爆炸。

### 7.3 DFS 枚举逻辑

函数大意：

```c
void dfsPaths(GameState state, int depth, int maxDepth, Path curPath);
```

注意：DFS 中每走一步都要调用 `simulateMove`，因为蛇身会动态变化，不能只看静态地图。

递归逻辑：

```text
如果当前蛇头到达食物：
    保存当前路径为候选路径
    return

如果 depth == maxDepth:
    return

枚举 W/A/S/D：
    如果不是反向，并且 simulateMove 后不死：
        递归搜索
```

为了减少爆炸，可以按“更接近食物”的方向优先搜索。

例如四个方向根据曼哈顿距离排序：

```text
优先尝试使 |newR-foodR| + |newC-foodC| 更小的方向
```

### 7.4 防止路径内部绕圈

可以适当避免重复访问同一位置，但不能太严格，因为蛇尾移动后某些格子会重新可用。

简单版可以记录当前路径中蛇头走过的位置：

```text
如果某个位置已经在当前路径中出现过，降低优先级或跳过
```

但第一版可以不做这个优化，直接靠最大深度限制。

## 8. 候选路径评分

每条候选路径走完后，得到一个吃掉当前食物后的状态 `S'`。

对 `S'` 评分。

### 8.1 评分目标

好的状态应该满足：

```text
1. 吃到了当前食物；
2. 蛇头所在区域可达空间大；
3. 蛇头能到达蛇尾附近；
4. 蛇不要被困在小空间；
5. 尽量少走路，减少自然增长带来的压力。
```

### 8.2 推荐评分函数

```text
score =
    + 10000                      // 成功吃到食物
    - 5 * pathLength             // 路径越短越好
    + 8 * reachableSpace         // 可达空间越大越好
    + 1000 * canReachTail        // 能追到尾巴很重要
    - 3000 * trappedPenalty      // 被困惩罚
```

其中：

#### `pathLength`

当前候选路径长度。

越短越好，因为每走 N 步会自然增长一次。

#### `reachableSpace`

从新蛇头出发，用 BFS 计算能到达多少格。

身体、墙、障碍视为不可走。

#### `canReachTail`

从新蛇头 BFS，看能否到达蛇尾位置。

```text
能到蛇尾：1
不能到蛇尾：0
```

为什么重要？

因为蛇尾代表未来会释放的空间。能追到尾巴，通常说明蛇还有生存通道。

#### `trappedPenalty`

如果：

```text
reachableSpace < len + 5
```

说明蛇当前活动空间可能不足，给大惩罚：

```text
trappedPenalty = 1
```

否则：

```text
trappedPenalty = 0
```

## 9. 没有安全吃食物路径时的保命策略

有时当前食物不可达，或者所有吃食物路径吃完后都很危险。

这时不要硬吃，改用保命策略。

优先级：

```text
1. 尝试走向蛇尾；
2. 如果无法走向蛇尾，选择使可达空间最大的方向；
3. 如果所有方向都会死，随便输出一个方向，等待结束。
```

### 9.1 追尾策略

用 BFS 找蛇头到蛇尾的路径。

如果找到路径，执行第一步。

因为尾巴会移动，追尾通常是比较安全的拖延方式。

### 9.2 最大空间策略

枚举四个方向：

```text
模拟走一步
如果不死：
    计算新蛇头 reachableSpace
选择 reachableSpace 最大的方向
```

## 10. 每一步主循环流程

主循环建议如下：

```text
while (true):

    1. 根据当前状态选择方向 dir：
        a. BFS 求蛇头到食物最短距离 d
        b. 如果 d 存在：
            DFS 枚举长度 <= d + DELTA 的安全候选路径
            对候选路径评分
            如果有高质量候选路径：
                dir = 最佳路径第一步
            否则：
                dir = 保命策略
        c. 如果 d 不存在：
            dir = 保命策略

    2. 输出：
        dirChar[dir]
        当前 score

    3. fflush(stdout)

    4. 保存 oldState = currentState

    5. 在本地状态中模拟执行这一步：
        simulateMove(&currentState, dir)

    6. 读取交互程序返回：
        r c

    7. 如果 r == 100 && c == 100：
        输出 oldState 的地图和 oldState 的 score
        break

    8. 如果 r > 0 && r < 19 && c > 0 && c < 19：
        表示刚才吃到了食物，新食物生成在 r c
        更新 currentState.foodR = r
        更新 currentState.foodC = c
        rebuildMap(&currentState)

    9. 如果 r == 20 && c == 20：
        正常继续，不更新食物
```

注意第 4、7 步：  
因为题目要求游戏结束时输出“碰撞前一刻”的地图，所以输出前要保存 `oldState`。如果本地模拟这一步会死，可以不更新状态；收到 `100 100` 后输出旧地图。

## 11. 地图更新规则

每次移动后建议重新生成地图，而不是手动局部改。

保留墙和障碍物：

```text
# 和 O 保持不变
其他位置先清成 .
再画蛇
再画食物
```

伪代码：

```c
void rebuildMap(GameState *s) {
    for r in 0..19:
        for c in 0..19:
            if baseMap[r][c] == '#':
                s->map[r][c] = '#';
            else if baseMap[r][c] == 'O':
                s->map[r][c] = 'O';
            else:
                s->map[r][c] = '.';

    for i = len - 1 down to 1:
        map[snake[i].r][snake[i].c] = 'B';

    map[snake[0].r][snake[0].c] = 'H';

    if food exists:
        map[foodR][foodC] = 'F';
}
```

最好单独保存一份 `baseMap`：

```c
char baseMap[20][21];
```

里面只保存：

```text
# 墙
O 障碍物
. 其他
```

## 12. 需要特别注意的坑

### 12.1 每次输出后必须刷新

```c
fflush(stdout);
```

交互题不刷新可能直接卡死。

### 12.2 输出的是移动前分数

题目要求每次交互输出方向和“蛇移动前的得分”。

所以顺序是：

```text
先输出当前 score
再模拟移动
再根据是否吃食物更新 score
```

### 12.3 自然增长和吃食物同一步只增长一节

写成：

```c
grow = eatFood || (nextStep % N == 0);
```

不要写成加两节。

### 12.4 普通移动时尾巴可以让出

如果本步不增长，蛇头走到当前尾巴格通常是允许的。

如果本步增长，尾巴不让出，不能走到尾巴格。

### 12.5 不能反向移动

例如当前向右，不能直接向左；否则游戏结束。

### 12.6 收到 `100 100` 后输出碰撞前地图

不是输出已经撞上后的地图。

## 13. 推荐实现顺序

不要一口气写完整复杂算法，按下面顺序实现。

### 第一阶段：基础状态维护

完成：

```text
读地图
恢复蛇数组
恢复当前方向
输出地图
simulateMove
rebuildMap
```

先用固定方向测试蛇能不能正常动。

### 第二阶段：安全方向选择

实现：

```text
枚举 W/A/S/D
选择第一个不会死的方向
```

保证程序不会轻易崩。

### 第三阶段：BFS 到食物

实现：

```text
BFS 找当前食物最短路
沿最短路走
如果没有路，走安全方向
```

这是第一个真正能拿分的版本。

### 第四阶段：候选路径 DFS

实现：

```text
BFS 求 d
DFS 枚举长度 <= d + DELTA 的安全路径
保存候选路径
```

### 第五阶段：评分函数

实现：

```text
reachableSpace
canReachTail
pathLengthPenalty
trappedPenalty
```

选择评分最高路径。

### 第六阶段：保命策略

实现：

```text
追尾
最大可达空间
```

## 14. 一句话总结算法

本程序不要只追求当前食物的最短路径，而是：

```text
先用 BFS 找到当前食物的最短距离，
再用 DFS 枚举若干条接近最短的安全路径，
逐条模拟蛇移动后的状态，
用“可达空间、能否追尾、路径长度、困死风险”给状态打分，
选择评分最高路径的第一步；
如果当前不适合吃食物，则优先追尾或走向最大安全空间。
```

这个方案的优势是：**比纯贪心安全，比全局搜索简单，比只用 BFS 更有未来意识**。
